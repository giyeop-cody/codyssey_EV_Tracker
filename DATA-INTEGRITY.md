## 2. 구조적 원인 5분류

1. **수집 지연** — 30분 주기 + GitHub 스케줄러 지연. 방금 생긴 평가는 최장 1~2시간 뒤에야 보일 수 있다(4스트림으로 완화했지만 0은 아님).
2. **표시 계층의 선택적 노출** — 캘린더 셀에는 완료(✔)/취소(✖)만 표시되고 진행/요청은 날짜 클릭 모달 뒤에 있었다. CANCELLED는 정책상 화면 제외. 겉으로는 "없음"과 구분 불가.
3. **요청단계(평가자 미상) 행의 시각 의미** — `s-` 접두 합성 행은 슬롯 시각이 아니라 피평가자 과제가 진행 상태로 바뀐 시각(`evlBgngDt`)을 쓴다. 실제 평가 슬롯은 항상 :00/:30이라, 17:39 같은 시각은 "가짜 평가"처럼 보여 혼란 유발.
4. **소스 시스템의 상태 지연** — 세션(슬롯)이 끝나도 소스가 완료로 전환하기 전까지 API는 "진행"을 돌려준다. 트래커는 미러이므로 그 상태를 그대로 노출한다 (예: 21:30~22:00 세션이 22:25 수집에도 진행).
5. **클라이언트 참조 시차** — 자동갱신(5분)은 탭이 보일 때만 동작한다. 오래 열어 둔 탭은 구버전 JSON을 보고 있을 수 있다.

## 3. 신고 접수 시 삼각측량 절차 (5단계)

1. **최신본 확인** — 라이브 `data/YYYY-MM.json`의 `meta.generatedAt` (7/20 22:25 신고 대상본은 정상 최신이었다).
2. **수집 여부 판정** — 해당 mbrId+과목으로 JSON 이벤트 검색. 없으면 수집 문제(세션 만료·스케줄러·API 변동), 있으면 표시 문제.
3. **첫 진입 시각** — `git log -S "<evalId>" --reverse -- docs/data/YYYY-MM.json`으로 데이터 진입 커밋 확인 → "신고 시점에 진짜 없었나" 판정.
4. **상태 매핑 확인** — 소스가 진행/완료 중 뭘 반환하는지 (완료 대기 상태인 건을 완료로 바꿔 칠하지 않는다).
5. **그래도 다르면** — 브라우저 강력 새로고침, 월 네비게이션, 히트맵 필터 칩 점검.
